Rendimiento y escalabilidad

Las métricas permiten entender qué tan rápido responde un sistema y qué tan bien puede crecer cuando aumenta la carga.

Voz: Español del navegador

En un sistema real (por ejemplo, una plataforma académica, una app de domicilios o una API de inventario), no basta con que el software “funcione”. También debe responder en un tiempo razonable y soportar el número de usuarios esperado. Cuando el sistema se vuelve lento o se cae, el problema casi siempre se relaciona con dos ideas: rendimiento (qué tan rápido responde) y escalabilidad (qué tan bien soporta más carga).

Para mejorar un sistema sin adivinar, se necesitan métricas: datos simples que nos muestran qué está pasando (tiempos, errores, uso de recursos). En esta guía verás qué métricas básicas se usan en arquitectura, dónde se miden y cómo se conectan con estrategias de escalabilidad en la nube.

Desarrollo del contenido

Voz: Español del navegador

En arquitectura, el rendimiento se observa para garantizar que el sistema responda bien a los usuarios, y la escalabilidad se planea para que el sistema pueda crecer sin “romperse”. La evaluación se apoya en métricas porque permiten comparar: antes vs. después de un cambio, horas normales vs. horas pico, o una versión vs. otra.

Un principio básico: primero se mide, luego se decide. Si una API responde lento, puede ser por la red, por el servidor, por la base de datos o por la forma en que está escrito el código. Las métricas ayudan a ubicar el problema en el punto correcto.

Idea práctica: en servicios web, el escalado horizontal funciona mejor cuando el servicio es “sin estado” (stateless), es decir, que no guarda información del usuario dentro de la misma instancia. Por eso se recomienda guardar datos en bases de datos o servicios externos.

Práctica superficial: observar tiempo de respuesta

El objetivo de la práctica no es hacer pruebas profesionales de carga, sino aprender a observar y registrar una métrica básica (tiempo de respuesta).

Puedes hacerlo de dos formas sencillas: Opción A: Navegador: abre un endpoint (por ejemplo, /salud o /productos) y revisa en “Network” cuánto tarda la respuesta. Opción B: Postman: envía una petición GET y observa el campo “Response Time”. Repite 5 veces y calcula un promedio simple.

Registro mínimo recomendado: anota el endpoint probado, el tiempo de respuesta en milisegundos (ms) y si hubo errores. Con esa información, propone una mejora (por ejemplo, cache o escalado).

Simulador de observación

Selecciona un endpoint y un escenario de carga para estimar tiempo de respuesta, tasa de error y posible mejora.

Guía rápida

Tiempo de respuesta: cuánto tarda en contestar la petición.

Tasa de error: ayuda a saber si el servicio empieza a fallar bajo presión.

Uso de recursos: si CPU o RAM se acercan al límite, puede haber saturación.

La meta es mirar la métrica, sospechar dónde está el cuello de botella y proponer una mejora básica.
