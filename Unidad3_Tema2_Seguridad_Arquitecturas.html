Seguridad en arquitecturas

La seguridad en arquitectura se diseña desde el inicio y define cómo se protege el sistema, los datos y a los usuarios.

Voz: Español del navegador

En un sistema real, la seguridad no es un “extra” que se agrega al final: es un conjunto de decisiones y controles que protegen el sistema, los datos y a los usuarios. Cuando una API expone información sensible, cuando un usuario no autorizado puede modificar datos, o cuando el sistema se cae por ataques o fallas, casi siempre hay un problema de arquitectura y de diseño.

En esta guía verás el rol de la seguridad en arquitectura, el enfoque de security by design y, como parte práctica mínima, cómo se relacionan la autenticación y la autorización en un servicio tipo API.

Esta guía está diseñada para un nivel básico de Tecnólogo. El foco principal es entender cómo la seguridad se integra en una arquitectura de sistema y qué decisiones se toman “desde el diseño”. La parte práctica es intencionalmente por encima: se trabaja el ciclo mínimo de autenticación (obtener un token) y autorización (permitir o negar acceso), sin profundizar en criptografía avanzada ni en configuraciones complejas de infraestructura.

Desarrollo del contenido

Voz: Español del navegador

En arquitectura, la seguridad se entiende como “cómo protegemos el sistema” en diferentes puntos: el cliente (app), la red (internet), el servicio (API) y los datos (base de datos). No se trata solo de una herramienta; se trata de decisiones: qué datos se exponen, quién puede acceder y qué pasa si algo falla.

Tres objetivos básicos de seguridad se resumen en la tríada CIA.

Figura 1. Ubicación de controles de seguridad en una arquitectura simple

Lectura de la figura: el cliente se comunica por HTTPS; la API valida identidad (autenticación) y permisos (autorización); la base de datos protege los datos; y el monitoreo registra eventos y alertas. En arquitectura, la seguridad se reparte en capas (defensa en profundidad).

Security by design significa pensar la seguridad desde el inicio del proyecto, no “parchar” al final. En un sistema básico, esto se logra con principios simples.

Ciclo de vida con security by design

1) Requisitos: ¿qué datos son sensibles? ¿quién debe acceder?

2) Diseño: ¿dónde pondré autenticación y autorización? ¿qué endpoints se protegen?

3) Implementación: validaciones, manejo de errores, control de acceso.

4) Pruebas: probar casos “malos” (sin token, token inválido, rol incorrecto).

5) Despliegue/operación: logs, actualización de dependencias y monitoreo.

Práctica superficial: probar acceso con y sin token

Propósito: comprender, con un caso real y sencillo, la diferencia entre autenticación (quién eres) y autorización (qué puedes hacer) en una API.

Requisitos: Node.js instalado y un cliente para probar endpoints (navegador, Postman o curl).

Escenario: “Inventario escolar”. Tendrás una API con estos endpoints: GET /salud, GET /productos, POST /login, GET /perfil y DELETE /productos/:id.

Simulador de acceso

Selecciona un endpoint y un tipo de token para ver qué respondería la API.

Guía rápida

401: no estás autenticado. Falta el token o es inválido.

403: sí estás autenticado, pero no tienes permisos suficientes.

200: la solicitud fue aceptada y el recurso puede consultarse o ejecutarse.

Paso 1. Crear la API de ejemplo: ejecuta npm init -y, luego npm install express, crea app.js y configura autenticación, autorización por rol y los endpoints del ejercicio.

Paso 2. Usa este simulador para anticipar qué debería pasar antes de probarlo en Postman, navegador o curl.

Situación: una app móvil consume una API de inventario. La API se conecta a una base de datos y registra logs. Actividad 2. Matriz de roles y permisos

Situación: tu API maneja productos. Tiene endpoints: GET /productos, POST /productos, PUT /productos/:id, DELETE /productos/:id. Define 3 roles (por ejemplo: Admin, Editor, Consulta) y completa una matriz indicando qué endpoints puede usar cada rol. Luego explica en 5 a 7 líneas por qué aplicaste mínimo privilegio.

Aciertos: 0/3

¿Qué rol es suficiente si una persona solo debe consultar productos?

Si un rol puede crear y actualizar productos, pero no borrarlos, ¿cuál es la mejor opción?

¿Qué principio respalda asignar solo los permisos estrictamente necesarios?

Actividad 3. Mini práctica de autenticación y autorización

Piensa en la práctica superficial de la API y responde según el comportamiento esperado con y sin token.

¿Qué representa el encabezado Authorization: Bearer token_user_123?

¿Cuál usuario del ejemplo obtiene el token de administrador?
