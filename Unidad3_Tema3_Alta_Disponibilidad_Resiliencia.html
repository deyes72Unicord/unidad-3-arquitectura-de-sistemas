Alta disponibilidad y tolerancia a fallos

La arquitectura resiliente busca que el servicio continúe disponible incluso cuando alguno de sus componentes falla.

Voz: Español del navegador

En sistemas reales (plataformas académicas, comercio electrónico, APIs de inventario), no solo importa que la aplicación funcione, sino que esté disponible cuando los usuarios la necesitan. Una caída en horas pico puede afectar matrículas, ventas, o la operación diaria de una institución.

Para reducir caídas, la arquitectura incorpora estrategias de continuidad: evitar puntos únicos de falla, agregar redundancia, definir failover, y preparar backups y planes de recuperación. En esta guía verás estos conceptos con vocabulario sencillo y un enfoque arquitectónico.

Desarrollo del contenido

Voz: Español del navegador

En arquitectura, alta disponibilidad (HA) significa diseñar el sistema para que esté “casi siempre” disponible. Tolerancia a fallos significa que, si una parte falla, el sistema sigue funcionando (total o parcialmente) sin detenerse por completo. Resiliencia es la capacidad de resistir fallas y recuperarse rápido.

En un nivel básico, el objetivo es claro: reducir el impacto de fallas comunes (un servidor que se apaga, una aplicación que se cae, una base de datos que no responde) para que el usuario final no pierda el servicio.

Figura 1. Arquitectura simple (con riesgo) vs arquitectura con redundancia (más disponible).

Un SPOF (Single Point ofFailure) es un componente que, si falla, detiene el servicio completo. Ejemplos típicos: una sola instancia de la API, una única base de datos, un único servidor de archivos, o una única conexión a internet.

La idea básica para mejorar es la redundancia: tener más de una opción. Por ejemplo, dos instancias de la API (API 1 y API 2) para que, si una cae, la otra atienda solicitudes.

Failover es el cambio a un “reemplazo” cuando algo falla. Por ejemplo, si la API 1 deja de responder, el sistema debe enviar el tráfico a la API 2. Muchas arquitecturas logran esto usando un balanceador de carga (load balancer) que reparte solicitudes y evita enviar tráfico a instancias caídas (healthchecks).

Nota: en este curso no buscamos configurar un balanceador real; lo importante es entender la función dentro de la arquitectura.

Figura 2. Balanceo + failover (concepto)

Un backup es una copia de seguridad de datos (por ejemplo, de la base de datos o archivos). Sirve para recuperar información si ocurre un error humano, daño, borrado accidental o ataque. DisasterRecovery (DR) es el plan para recuperar el sistema completo (servicio + datos) después de un incidente mayor.

Dos conceptos básicos para planear recuperación: RPO (Recovery Point Objective): cuánta información se puede “perder” como máximo. RTO (Recovery Time Objective): cuánto tiempo máximo se acepta estar caído.

A nivel introductorio, una arquitectura más resiliente suele incluir: redundancia de servicios críticos, backups programados y pruebas de restauración, monitoreo básico y documentación simple sobre qué hacer cuando ocurre una falla.

Guía rápida

SPOF: si falla, tumba todo el servicio.

Failover: cambia el tráfico a una instancia sana.

Backup: recupera datos.

DR: recupera el sistema completo después de un incidente mayor.
